 Welcome to Jeremy's IT Lab. This is a complete course for the CCNA. In this video, we will cover a topic related to virtualization that is Containers. Containers are similar to VMs, virtual machines. However, there are some key differences that you must understand. If you are interested in the field of DevOps, you will have to learn Containers in Depth. For the CCNA, however, you only need to understand the very basics. Here's what we'll cover in this video. First, we will review virtual machines. Containers are a similar technology to virtual machines, so it's important to understand both so we can compare them. Then we will look at Containers. You may have heard of Docker before, the most famous Container Engine available. Finally, we will spend some time comparing virtual machines and containers. To begin, here's an example of apps running on a server without virtualization. There is a single operating system, for example, Windows Server or Red Hat Linux, and on top of that OS, all of the apps are running. The apps are not isolated. A problem in one app could affect all of the other apps. Now you could isolate them by buying a separate physical server for each individual app, but that is extremely cost inefficient. Virtual machines are a solution to this problem. Virtual machines, VMs, allow multiple operating systems to run on a single physical server, and then apps are installed on those operating systems. A hypervisor is used to manage and allocate hardware resources to each VM. Remember, the VMs are all running on top of the same physical hardware, so they need a hypervisor to allocate those resources appropriately. There are two main types of hypervisors. Type 1 hypervisors, also known as native or bare metal hypervisors, run directly on top of hardware. That is the top diagram. Notice the Type 1 hypervisor directly on top of the hardware, with each VM then running on top of the hypervisor. Type 2 hypervisors, also known as hosted hypervisors, run on top of a host OS, for example Windows or perhaps Mac OS or some kind of Linux. That is shown in the bottom diagram. On top of the hardware there is a host OS, with the Type 2 hypervisor and its VMs installed on top of the host OS. Type 1 hypervisors are widely used in data center environments. If an enterprise has servers installed in a data center and is running its apps on VMs in those servers, you can be sure those VMs are running on a Type 1 hypervisor. Type 2 hypervisors, on the other hand, are commonly used on personal devices. For example, running a virtual network lab on your PC using Cisco modeling labs, CML. For my personal studies, I use CML to run virtual labs. Windows is installed on my PC hardware, I run VMware Workstation, a Type 2 hypervisor in Windows, and then CML runs as a VM in Workstation. Note that the OS in each VM can be the same or different. If you are running multiple VMs, one can be running Windows, the other Linux, and the other Mac OS. That's no problem. The binaries and libraries that I have indicated in gray in the diagrams are software libraries and services needed by the apps running in each VM. A VM allows its app or apps to run in an isolated environment, separate from the apps in other VMs. If there is an issue with one app or one of the guest OS's, it's not going to affect the others. And VMs are easy to create, delete, move, etc. For example, a VM can be easily saved and moved between different physical servers. Here's a very simple diagram to help you visualize the difference between Type 1 and Type 2 hypervisors. Type 1 runs directly on hardware, and Type 2 runs on a host OS. Virtualization using VMs allows much more efficient use of server hardware, and VMs are widely in use today. However, containers are a more recent virtualization technology being used more and more in recent days. Let's move on to the main topic of today's video, containers, and then compare them to VMs. This diagram shows the basic architecture of how containers run. Containers are software packages that contain an app and all dependencies, the binaries and libraries in the diagram for the contained app to run. Note that multiple apps can be run in a single container, although this is not how containers are usually used. You can generally assume that one container means one app. Containers run on a container engine, for example Docker engine, which is the most popular one. And that container engine is run on a host OS, usually Linux, which runs on the hardware. Containers are lightweight and include only the dependencies required for the specific app. No need to run an OS in each container, unlike in VMs. That is the major difference between VMs and containers. VMs run an OS in each VM, whereas containers don't. All of the different costs and benefits of VMs versus containers come from that major difference. Now a container orchestrator is a software platform for automating the deployment, management, scaling, etc. of containers. For example, you might have heard of Kubernetes, which is the most popular container orchestrator. Docker also has one called Docker swarm. Now why would we need a container orchestrator? Well in small numbers, manual operation of containers is possible, but large scale systems, for example involving microservices, can require thousands of containers. That many containers cannot be realistically managed manually. Since I mentioned microservices, microservice architecture is an approach to software architecture that divides a larger solution into smaller parts called microservices. So instead of one monolithic app, you might have hundreds of different microservices running together to form the larger solution. And those microservices all run in containers that can be orchestrated by Kubernetes or another platform. So that's a very quick overview of containers. Now let's compare VMs and containers. As I mentioned before, these differences all stem from the fact that each VM runs its own OS, whereas containers all run on top of a shared OS. That major architectural difference has some implications that make containers favored in many cases these days. First of all, VMs can take minutes to boot up, whereas containers can boot up in milliseconds. This makes containers much more agile. For example, if one container crashes, it takes very little time for another to boot up and take over. And VMs take up more disk space, often tens of gigabytes, whereas containers usually only take tens of megabytes. Again, that stems from the fact that each VM includes its own OS, whereas a container only contains an app and its dependencies. And for the same reason, VMs use more CPU and RAM resources compared to containers. Regarding portability, VMs are quite portable and can move between physical systems running the same hypervisor, but containers are even more so. They are smaller, faster to boot up, and darker containers can be run on nearly any container service. Also with these points, containers are looking much better than VMs, but VMs do have their advantages. One advantage is that VMs are more isolated because each VM runs its own OS. An issue on one OS won't affect apps in other VMs. However, that can't be said for containers. If the OS all containers run on top of crashes, the containers are all affected. Note that this isolation also provides security benefits for VMs. To summarize, there is a major movement toward the use of containers, especially with the rise of microservices, automation, and DevOps, which is the combination of software development and IT operations. But VMs are still widely used today, and they aren't going to go away anytime soon. The discussion of VMs versus containers has been had many times online. Do a search on Google or YouTube, and you'll find lots of resources if you want to learn more. Here's what we covered in this video. First we briefly reviewed virtual machines, and then we covered containers and compared them with VMs. Containers and VMs are similar, but also different in key ways. There is, of course, a lot of depth to the topic of containers, Docker, Kubernetes, etc. For the CCNA, you should just know the basics I mentioned in this video. --- VIDEO 111: Free CCNA | VRF | Day 54 (part 3) | CCNA 200-301 Complete Course ---
Domain: 3-IP_Connectivity

 Welcome to Jeremy's IT Lab. This is a complete course for the CCNA. In this video, we will cover virtual routing and forwarding, VRF. VRF allows us to divide one physical router into multiple virtual routers. You can think of it like VLANs for routers. VLANs divide a switch into multiple virtual switches, each with their own broadcast domain. And VRF divides a router into multiple virtual routers, each with their own routing table. Here's what we'll cover in this video. First, I will introduce the concept of VRF and its capabilities. And then introduce VRF configuration. Note that VRF configuration isn't actually a CCNA exam topic. But I think seeing the configurations will help you understand how it works. Unfortunately, VRF isn't supported in packet tracer. So if you want to try it in a lab, you'll need to use something like Cisco Modeling Labs, CML, or real Cisco devices. Let's get started. The diagram at the bottom gives an idea of how VRFs work. The black box in the middle represents R1, a single physical router. The blue, green, and red router icons inside represent three virtual routers inside R1, three VRFs. Let's call them VRF1, VRF2, and VRF3. So virtual routing and forwarding is used to divide a single router into multiple virtual routers. This is similar to how VLANs are used to divide a single switch, a LAN, into multiple virtual switches, virtual LANs. By default, all interfaces on a switch are in the same LAN, the same broadcast domain. When using VLANs, we can divide that one broadcast domain into multiple virtual broadcast domains. Similarly, by default, all router interfaces are all in the same routing domain. By routing domain, I mean, by default traffic received on one router interface, can be forwarded out of any other interface on the router if the routing table says to do so. For example, without using VRF, in the diagram below, traffic received on R1's G00 interface can be routed to and forwarded out of R1's G12 interface. However, with VRF, that won't be possible. G00 is in VRF1, and G12 is in VRF3, so traffic cannot be forwarded between them. How does VRF accomplish this? It does this by allowing a router to build separate routing tables. Normally, a router has one routing table, but with VRF, it has multiple. Interfaces, specifically layer three interfaces, are configured to be in a specific VRF, also known as a VRF instance. Keep in mind that VRF does not apply to layer two interfaces on a switch. Only router interfaces, SVIs, and routed ports on multi-layer switches can be configured in a VRF. It's a layer three concept. As I mentioned before, traffic in one VRF cannot be forwarded out of an interface in another VRF. However, as an exception, VRF leaking can be configured to allow traffic to pass between VRFs. That is a more advanced concept, though, and I won't cover it in this video. So what is the main purpose of VRF? Well, VRF is commonly used to facilitate MPLS, multi-protocol label switching. However, that's actually not the kind of VRF we are talking about here. The kind of VRF we are talking about is VRF light, VRF without MPLS. So keep in mind that when I say VRF in this video, I'm really referring to VRF light. VRF is commonly used by service providers to allow one device to carry traffic from multiple customers. How does it help with that? Well, there are two main ports. First, each customer's traffic is isolated from the others, because each customer connects to their own virtual router within the service provider's physical router. In the diagram below, let's say the blue, green, and red devices represent three different customers. All customers connect to the same service provider router, R1, but each connects to a different virtual router, a different VRF instance. And another major point is that customer IP addresses can overlap without issues. For example, the blue customer uses subnets 192.168.1.0 slash 24, and 192.168.2.0 slash 24. The green customer also uses 192.168.1.0 slash 24 and 2.0 slash 24. And so does the red customer. Without VRF, this kind of configuration is not possible on a router. But with VRF, it works fine, even though the IP addresses all overlap. Okay, now let's try configuring VRF to help understand how it works. To demonstrate VRF configuration, I'll use the network below. SPR1 is a service provider router, providing WAN services to two customers. Customer 1 and customer 2. C1R1 and C1R2 belong to customer 1 and C2R1 and C2R2 belong to customer 2. Note the subnets being used. Both customer 1 and customer 2 use subnet 192.168.1.0 slash 30. In this case, it's the exact same IP address. But even if the IP address is different, it won't work. Let me demonstrate that. To prove that point, I tried to configure it as 192.168.1.2, but again, it didn't work. Even if the IP address is different, G02 cannot be configured in the same subnet as G00. So without the use of VRF, two interfaces on the same router cannot be in the same subnet. Next, let's use VRF to configure this network properly. First, we have to create the VRFs, which can be done in global config mode with the command IP VRF, followed by the VRF name. Here, I created two VRFs, one named customer 1 and the other customer 2. I then use the command show IP VRF, which shows all of the VRFs on the router. As you can see, the customer 1 and customer 2 VRFs were created. The next step is to assign interfaces to each VRF. So from interface config mode for G00, I used the command IP VRF forwarding customer 1. G00 is now part of the customer 1 VRF. However, I also got this message. Interface G00 IP V4 disabled and addresses removed due to enabling VRF customer 1. If an interface has an IP address configured, the IP address will be removed when you assign the interface to a VRF. Show IP route displays the global routing table. That's the normal routing table you're used to when not using VRFs. Having four addresses like this isn't necessary in a wired Ethernet network, but 802.11 wireless networks have unique requirements. The next field is sequence control. It's used to reassemble fragments and eliminate duplicate frames. Then there is QOS control, which as you probably guessed is used in QOS to prioritize certain traffic. After that is a field called HT, high throughput control. This was added in 802.11N to enable high throughput operations. I didn't mention this in the previous video, but 802.11N, Wi-Fi 4, is also known as high throughput Wi-Fi. And after it, 802.11AC, Wi-Fi 5, is also known as very high throughput, VHT Wi-Fi. So HT control is the last field of the header, and after it is the frame body, which is the packet encapsulated within the 802.11 frame. And finally there is the FCS frame check sequence in the trailer. Just like in an Ethernet frame, this is used to check for errors in the frame. And that's all for 802.11 frames. Remember, not all of these fields are present in all 802.11 messages. It all depends on the version of 802.11 being used, and the message type. Like I said, these frames and their operations can be much more complicated than Ethernet frames. If you want to be an expert in wireless networks, you should definitely learn all the details, but that's enough for now. Next, before 802.11 message types, I want to introduce the 802.11 Association process, which I only briefly mentioned in the last video. Access points are used to bridge traffic between wireless stations and other devices, for example hosts connected to the wired network. But for a station to send traffic through the AP, it must be associated with the AP, and there is a process used to do that, to form that connection. There are three connection states. First, when the station is not authenticated or associated with the AP. Second, when the station is authenticated but not yet associated with the AP. And finally, when the station is authenticated and associated with the AP. The station must be authenticated and associated with the AP to send traffic through it. So let's look at a simple version of that process. First the station sends a probe request message to learn about what APs and BSSs are available, and the AP sends a probe response to say that it is available. Note that there are actually two ways for a station to scan for a BSS. First is active scanning, where the station sends probe requests and listens for a probe response from an AP. That's what I show in the diagram on the left. But there is also passive scanning, in which the station listens for beacon messages from an AP. Beacons are sent periodically by APs to advertise the BSS. Okay, so using either active or passive scanning, the station knows about the AP and its BSS. However, it's still in that first connection state, not authenticated and not associated. Then there is an authentication exchange. For example, the station sends a password to the AP and the AP authenticates it. If this is successful, we have reached the second state, authenticated but not yet associated. Finally, there is an association request and response. And if this is successful, we have reached the final connection state, authenticated and associated. Finally, the station can send traffic through the AP. Now, all of these messages, probe, beacon, authentication and association are the same type of 802.11 message. Let's look at the different types. There are three 802.11 message types. First, there are management frames, which are used to manage the BSS. For example, the messages I introduced in the previous slide, beacon, probe, authentication and association are all management frames. There are more management messages than these, but I won't introduce them all. The second type is control. These frames are used to control access to the medium, the radio frequency. They assist with the delivery of management and data frames. For example, request to send and clear to send messages, which I very briefly mentioned in the previous video are control messages. Also, AC messages, used to acknowledge that a message was received are control frames. The third type of message is data. These are the messages used to send actual data packets. First up, we are going to talk about different ways of deploying, of using wireless access points. There are three main wireless AP deployment methods, autonomous, lightweight and cloud-based. I'll introduce autonomous APs first. Autonomous APs are self-contained systems that don't rely on a WLC, wireless LAN controller. Hence the name autonomous, they are independent. And that means they are configured individually. They can be configured by console cable, remotely via telnet or SSH, or also an HTTP or HTTPS web connection via a web browser GUI. For small networks, this is fine. But configuring APs individually in a large network quickly becomes unrealistic. Note that an IP address for remote management should be configured on the APs, so you can connect via telnet SSH or HTTP to configure them. Radio frequency parameters, such as how much power it should use to transmit, and which channel it should use, must be configured manually per AP. Security policies, for example, ACLs, are also handled individually by each AP. Other settings such as QOS are also configured per AP, and there is no central monitoring or management of the APs. So as I said before, each AP is its own self-contained system. Here's an example network with autonomous APs. When using autonomous APs, each AP should connect to the wired network with a trunk link. In this case, there are two SSIDs, so it might be obvious that a trunk connection is needed, because each SSID is mapped to a VLAN. But even if the AP provides only one SSID, the connection should be a trunk. Why is that? It's because the management traffic used to remotely connect to the wireless APs, as well as other devices, the switches, should be in a separate VLAN. I might not have mentioned this earlier in the course, but it's best practice to keep management traffic separate from regular data traffic by putting it in a separate subnet and VLAN. So there are trunk links between each of these switches carrying VLANs 10 and 20 for the wireless clients, as well as VLAN 99 for management traffic. When using autonomous APs, data traffic from wireless clients has a very direct path to the wired network, or to other wireless clients associated with the same AP. This statement might be a little unclear, but it'll understand it when you see the path that data takes when using a wireless LAN controller. So with an autonomous AP, traffic between these PCs simply goes from PC to AP, among the switches, and then to the other AP and PC. Or between devices associated with the same AP, there's no need for the traffic to even go to the wired network. Soon you'll see how this is different compared to the traffic path when using a wireless LAN controller. Notice that each VLAN has to stretch across the entire local area network. This is considered a bad practice. Why is that? Why is it bad to have the same VLANs stretched across the whole network? Well, there are many reasons, but here are a few. First, large broadcast domains. Each VLAN is a broadcast domain, and if the VLANs are stretched across the whole network like this, each broadcast message will be flooded throughout the whole network. The second reason is that spanning tree will disable links. A big focus of modern network design is avoiding spanning tree as much as possible, because disabling links means a reduction in total bandwidth. Also, adding and deleting VLANs is very labor intensive, if it has to be done over dozens of switches in a large network. So to summarize, autonomous APs can be used in small networks, but they are not viable in medium to large networks. A large network can have thousands of APs, and configuring and managing each of those APs one by one is not realistic. Finally, remember that autonomous APs can also function in the modes covered in the previous video, Repeater, Outdoor Bridge, or Workgroup Bridge. Now let's talk about the next major deployment model for APs, Lightweight APs. The functions of an AP can be split between the AP and a wireless LAN controller, WLC. The Lightweight APs handle real-time operations like transmitting and receiving radio frequency traffic, encryption and decryption of traffic, sending out beacons and probes, etc. However, other functions are carried out by AWLC, for example RF management, security and QOS management, client authentication, client association and roaming management, etc. The WLC centrally controls all of these things for many lightweight APs. This is called split-mac architecture, split-media access control, because the functions are split between the lightweight APs and the WLC. Note that the WLC is also used to centrally configure the lightweight APs. No more need to log into the CLI of each AP and manually configure them one by one. It's all done centrally from the WLC. Note that the WLC can be located in the same subnet and VLAN as the lightweight APs it manages, or in a different subnet and VLAN. I'll talk about different WLC deployments later. Note that the WLC and lightweight APs authenticate each other using digital certificates installed on each device. These certificates follow the X509 standard, which is the same that websites use to prove their authenticity over the internet. This ensures that only authorized APs can join the network, so an attacker can't make their AP join the network. Here's an example network with a wireless LAN controller added this time. The WLC and lightweight APs use a protocol called capwap, control and provisioning of wireless access points to communicate. Capwap is based on an older protocol called LWAPP, lightweight access point protocol. For communications, two tunnels are created between each AP and the WLC, like this. Note that each of those tunnels in the diagram represents two separate tunnels, each used for different purposes. So let's see what those two tunnels are. One is the control tunnel, and it uses UDP port 5246. There's another port number for you to remember. Capwap control tunnels use UDP port 5246. So this tunnel is used to configure the APs and control and manage their operations. Note that all traffic in this tunnel is encrypted by default, so it is secure. The second tunnel is the data tunnel using UDP port 5247. All traffic from wireless clients is sent through this tunnel to the WLC. It does not go directly to the wired network, or even directly to another client associated with the same AP. It must go through the tunnel to the WLC first. Now, of course, in reality, the traffic does pass through the wired network to go to the WLC. This isn't a complete list and you don't have to memorize it, but you should be aware of what benefits are provided. First, scalability. With a WLC or even multiple WLCs and large networks, it's much simpler to build and support a network with thousands of APs. That would simply not be feasible with autonomous APs. WLCs also provide dynamic channel assignment. The WLC can automatically select which channel each AP should use, so you don't have to manually plan with channel each will use. The WLC can also automatically set the appropriate transmit power for each AP to provide sufficient coverage without interfering with other APs. The WLC can also provide self-healing wireless coverage. So when an AP stops functioning, the WLC can increase the transmit power of nearby APs to avoid coverage holes. Seamless roaming is also provided, so clients can roam between APs with no noticeable delay. Another benefit is client load balancing. So if a client is in range of two APs, the WLC can associate the client with the least used AP to balance the load among APs. And the last benefit I'll mention is security and QOS management. Central management of security and QOS policies ensures consistency across the network.
