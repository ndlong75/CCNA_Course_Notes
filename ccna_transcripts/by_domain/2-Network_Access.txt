For example, the station sends a password to the AP and the AP authenticates it. If this is successful, we have reached the second state, authenticated but not yet associated. Finally, there is an association request and response. And if this is successful, we have reached the final connection state, authenticated and associated. Finally, the station can send traffic through the AP. Now, all of these messages, probe, beacon, authentication and association are the same type of 802.11 message. Let's look at the different types. There are three 802.11 message types. First, there are management frames, which are used to manage the BSS. For example, the messages I introduced in the previous slide, beacon, probe, authentication and association are all management frames. There are more management messages than these, but I won't introduce them all. The second type is control. These frames are used to control access to the medium, the radio frequency. They assist with the delivery of management and data frames. For example, request to send and clear to send messages, which I very briefly mentioned in the previous video are control messages. Also, AC messages, used to acknowledge that a message was received are control frames. The third type of message is data. These are the messages used to send actual data packets. First up, we are going to talk about different ways of deploying, of using wireless access points. There are three main wireless AP deployment methods, autonomous, lightweight and cloud-based. I'll introduce autonomous APs first. Autonomous APs are self-contained systems that don't rely on a WLC, wireless LAN controller. Hence the name autonomous, they are independent. And that means they are configured individually. They can be configured by console cable, remotely via telnet or SSH, or also an HTTP or HTTPS web connection via a web browser GUI. For small networks, this is fine. But configuring APs individually in a large network quickly becomes unrealistic. Note that an IP address for remote management should be configured on the APs, so you can connect via telnet SSH or HTTP to configure them. Radio frequency parameters, such as how much power it should use to transmit, and which channel it should use, must be configured manually per AP. Security policies, for example, ACLs, are also handled individually by each AP. Other settings such as QOS are also configured per AP, and there is no central monitoring or management of the APs. So as I said before, each AP is its own self-contained system. Here's an example network with autonomous APs. When using autonomous APs, each AP should connect to the wired network with a trunk link. In this case, there are two SSIDs, so it might be obvious that a trunk connection is needed, because each SSID is mapped to a VLAN. But even if the AP provides only one SSID, the connection should be a trunk. Why is that? It's because the management traffic used to remotely connect to the wireless APs, as well as other devices, the switches, should be in a separate VLAN. I might not have mentioned this earlier in the course, but it's best practice to keep management traffic separate from regular data traffic by putting it in a separate subnet and VLAN. So there are trunk links between each of these switches carrying VLANs 10 and 20 for the wireless clients, as well as VLAN 99 for management traffic. When using autonomous APs, data traffic from wireless clients has a very direct path to the wired network, or to other wireless clients associated with the same AP. This statement might be a little unclear, but it'll understand it when you see the path that data takes when using a wireless LAN controller. So with an autonomous AP, traffic between these PCs simply goes from PC to AP, among the switches, and then to the other AP and PC. Or between devices associated with the same AP, there's no need for the traffic to even go to the wired network. Soon you'll see how this is different compared to the traffic path when using a wireless LAN controller. Notice that each VLAN has to stretch across the entire local area network. This is considered a bad practice. Why is that? Why is it bad to have the same VLANs stretched across the whole network? Well, there are many reasons, but here are a few. First, large broadcast domains. Each VLAN is a broadcast domain, and if the VLANs are stretched across the whole network like this, each broadcast message will be flooded throughout the whole network. The second reason is that spanning tree will disable links. A big focus of modern network design is avoiding spanning tree as much as possible, because disabling links means a reduction in total bandwidth. Also, adding and deleting VLANs is very labor intensive, if it has to be done over dozens of switches in a large network. So to summarize, autonomous APs can be used in small networks, but they are not viable in medium to large networks. A large network can have thousands of APs, and configuring and managing each of those APs one by one is not realistic. Finally, remember that autonomous APs can also function in the modes covered in the previous video, Repeater, Outdoor Bridge, or Workgroup Bridge. Now let's talk about the next major deployment model for APs, Lightweight APs. The functions of an AP can be split between the AP and a wireless LAN controller, WLC. The Lightweight APs handle real-time operations like transmitting and receiving radio frequency traffic, encryption and decryption of traffic, sending out beacons and probes, etc. However, other functions are carried out by AWLC, for example RF management, security and QOS management, client authentication, client association and roaming management, etc. The WLC centrally controls all of these things for many lightweight APs. This is called split-mac architecture, split-media access control, because the functions are split between the lightweight APs and the WLC. Note that the WLC is also used to centrally configure the lightweight APs. No more need to log into the CLI of each AP and manually configure them one by one. It's all done centrally from the WLC. Note that the WLC can be located in the same subnet and VLAN as the lightweight APs it manages, or in a different subnet and VLAN. I'll talk about different WLC deployments later. Note that the WLC and lightweight APs authenticate each other using digital certificates installed on each device. These certificates follow the X509 standard, which is the same that websites use to prove their authenticity over the internet. This ensures that only authorized APs can join the network, so an attacker can't make their AP join the network. Here's an example network with a wireless LAN controller added this time. The WLC and lightweight APs use a protocol called capwap, control and provisioning of wireless access points to communicate. Capwap is based on an older protocol called LWAPP, lightweight access point protocol. For communications, two tunnels are created between each AP and the WLC, like this. Note that each of those tunnels in the diagram represents two separate tunnels, each used for different purposes. So let's see what those two tunnels are. One is the control tunnel, and it uses UDP port 5246. There's another port number for you to remember. Capwap control tunnels use UDP port 5246. So this tunnel is used to configure the APs and control and manage their operations. Note that all traffic in this tunnel is encrypted by default, so it is secure. The second tunnel is the data tunnel using UDP port 5247. All traffic from wireless clients is sent through this tunnel to the WLC. It does not go directly to the wired network, or even directly to another client associated with the same AP. It must go through the tunnel to the WLC first. Now, of course, in reality, the traffic does pass through the wired network to go to the WLC. However, it is encapsulated with new headers to form the tunnel, just like I showed with GRE and IPsec tunnels in the WAN video. Note that traffic in this tunnel is not encrypted by default, but you can configure it to be encrypted using a protocol called DTLS, Datagram Transport Layer Security. I mentioned TLS, Transport Layer Security, briefly in the WAN video. DTLS is basically the same thing, but it uses UDP, whereas regular TLS uses TCP. Now, here's another difference between lightweight and autonomous APs. Because all traffic from wireless clients is tunneled to the WLC with CAPWAP, APs connect to switch access ports, not trunk ports. There should be a VLAN for each SSID the AP offers, as well as a VLAN for network device management. If a wireless client wants to send traffic out to an external network, it sends the frame to its wireless access point, which then forwards it to the default gateway over the wired network, which will then forward it according to its routing table. And if a client wants to send traffic to another device associated with the same AP, the traffic has no need to enter the wired network. The frame is sent to the AP and then simply forwarded to the destination. When using a SplitMac architecture, this is all very different. As I said before, the AP doesn't have to connect to the switch with a trunk, an access link is fine. However, a trunk link is needed to connect the WLC to the wired network. In this case, instead of the AP mapping each SSID to a VLAN and forming the border between the wired and wireless network, the WLC takes that role. Traffic from a wireless client is sent to the AP and then tunneled to the WLC. Then the WLC sends it over the wired network to the default gateway, which will forward it as appropriate. Even if the traffic is destined for a host associated with the same AP, it is first tunneled to the WLC and then tunneled back and sent to the destination. Okay, hopefully that helped demonstrate the differences between the traffic paths used by each of the two architectures. There are some key benefits to using a SplitMac architecture. This isn't a complete list and you don't have to memorize it, but you should be aware of what benefits are provided. First, scalability. With a WLC or even multiple WLCs and large networks, it's much simpler to build and support a network with thousands of APs. That would simply not be feasible with autonomous APs. WLCs also provide dynamic channel assignment. The WLC can automatically select which channel each AP should use, so you don't have to manually plan with channel each will use. The WLC can also automatically set the appropriate transmit power for each AP to provide sufficient coverage without interfering with other APs. The WLC can also provide self-healing wireless coverage. So when an AP stops functioning, the WLC can increase the transmit power of nearby APs to avoid coverage holes. Seamless roaming is also provided, so clients can roam between APs with no noticeable delay. Another benefit is client load balancing. So if a client is in range of two APs, the WLC can associate the client with the least used AP to balance the load among APs. And the last benefit I'll mention is security and QOS management. Central management of security and QOS policies ensures consistency across the network. As I said, you don't have to memorize this list, but definitely be aware that the split Mac architecture offers various benefits.
